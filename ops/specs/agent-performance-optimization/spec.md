# 명세: 에이전트 성능평가 및 성능 최적화와 도커 자동 재빌드 적용

- **slug**: agent-performance-optimization
- **헌법 항목**: III 검증, IV 공정성, V 학습·진화, VI 토큰 절약

## 문제 / 목표
현재 jeo-claw 시스템은 ZeroClaw와 NullClaw의 A/B 비교를 위한 메트릭 정의 및 대시보드 렌더링 뼈대(`compare/runner.ts`, `compare/metrics.ts`, `compare/dashboard.ts`)를 가지고 있으나, 실제 라이브 메트릭을 수집하여 성능을 평가하는 실행 명령이 부재합니다. 또한, 에이전트 실행 시 메모리 누수나 불필요한 리소스 낭비를 방지하는 최적화가 필요하며, 코드 변경 시 도커 컨테이너가 자동으로 재빌드되어 즉시 반영되는 환경이 구축되어야 합니다.

## 유저 스토리
- 운영자로서 에이전트의 성능(레이턴시, 메모리, CPU, 토큰 비용, 실패율)을 실시간으로 평가하고 최적화하기 위해, 실제 메트릭을 수집하고 대시보드를 출력하는 실행 명령을 원한다.
- 개발자로서 코드 변경 시 수동으로 도커를 재빌드하는 번거로움을 줄이기 위해, 코드 변경을 감지하여 도커 컨테이너를 자동으로 재빌드하고 재시작하는 자동화 도구를 원한다.

## 수용 기준 (measurable)
- [ ] `bun run compare` 명령을 통해 실제 또는 모의 에이전트 실행 메트릭을 수집하고 대시보드를 터미널에 출력할 수 있어야 함.
- [ ] 에이전트 실행 및 메트릭 수집 과정에서 메모리 누수가 없음을 검증하는 테스트 또는 모니터링 코드가 포함되어야 함.
- [ ] 코드 변경 시 도커 이미지를 자동으로 재빌드하고 컨테이너를 재시작하는 스크립트(`scripts/docker-watch.ts`)를 제공해야 함.
- [ ] 4대 정적 게이트 green(tsc/test/check:compose/validate)

## 비목표 (non-goals)
- 외부 클라우드 모니터링 서비스(Datadog 등)와의 연동은 범위 외로 함.
- ZeroClaw/NullClaw 자체 바이너리의 내부 C/Rust/Zig 코드 최적화는 범위 외로 하고, Bun/TS 오케스트레이션 레이어 및 메트릭 수집기의 최적화에 집중함.

## 보안·공정성 영향
- A/B 공정성 유지를 위해 ZeroClaw와 NullClaw의 메트릭 수집 방식은 동일해야 함.
- 도커 재빌드 스크립트는 로컬/개발 환경에서만 실행되며, 프로덕션 보안 설정을 약화시키지 않아야 함.
